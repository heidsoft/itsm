# ITSM 浏览器功能测试与改进建议计划

## Context

ITSM（对标 ServiceNow 的开源 ITSM，GitHub 31 stars）最近由 Codex 完成了工作流增强和 marketplace 新模块的提交（commit `ddb51e5a`），但相关功能在浏览器中是否真的可用、是否引入回归尚无验证。本计划的目标是：

1. **用浏览器实际测试当前 ITSM 系统功能**（核心模块 + 最近增强的工作流 + 新增 marketplace）
2. **提出下一步改进建议**，帮助项目从"功能实现"走向"开源产品"。

执行前已确认的运行事实：
- 后端 8090 健康（`GET /api/v1/health` → 200）
- 前端 3000 启动但 dev server 静态资源大量 404（`main-app.js`/`app-pages-internals.js`/`app/(main)/layout.js` 等），浏览器实际为白屏
- `admin/admin123` 可正常登录获取 JWT
- 后端 `Set-Cookie: access_token=...`，但前端 `token-storage.ts` 注释里写 `auth-token`（命名不一致）
- `next.config.ts` 仍保留 `ignoreBuildErrors:true` + `ignoreDuringBuilds:true`
- `GET /api/v1/marketplace/items` → **500**：`pq: relation "marketplace_items" does not exist`（commit 新增的 Ent 实体未跑 migration）
- 既有 6 份 review/测试报告（`docs/review/*.md`）已覆盖 UI 视觉、商业就绪度、ServiceNow 对标等，本次不重复

用户已确认：
- **测试范围**：工作流 + 工单/事件/问题/变更 + dashboard/CMDB/知识库 + marketplace
- **执行方式**：仅测后端 API（因前端 dev server 不可用）
- **交付物**：完整浏览器测试报告 markdown

## 实施步骤

### 阶段 0 — 环境就绪与已知问题清点（10 分钟）

不修复，仅记录。

- 验证 `:8090` / `:3000` 进程
- 验证 PostgreSQL / Redis 可达
- 列出 `marketplace_*` 表是否存在
- 列举 `itsm-backend/migrations/` 中是否有 `create_marketplace_*.sql`
- `curl -I` 验证 `_next/static/chunks/*.js` 的 404
- 记录 cookie 命名不一致（`access_token` vs `auth-token`）
- 输出：`output/functional-test/stage0_environment.txt`

### 阶段 1 — 基础设施（M0，15 分钟）

curl + jq 测试 18 个端点：health/version/readiness/ga、csrf-token、login/refresh/me/logout、tenants/menus、notifications（4）、connectors（3）、ws/ticket。重点验证：
- TC-1.2 登录返回字段名（`access_token`，不是 `auth-token`）
- TC-1.3 菜单去重（上次报告 P2 回归）
- TC-1.7 WS 票据一次性

输出：`stage1_infrastructure.json`

### 阶段 2 — 核心数据（M1-M4，30 分钟）

为后续 BPMN 测试准备真实数据，含上次报告回归点：

- **TC-2.1 工单 CRUD + 搜索 + stats**（回归 `q/keyword` 参数、`stats.open=0`、version 递增）
- **TC-2.4 工单工作流 8 核心 API**（回归 `workflow/accept` 字段 `ticketId`、`workflow-history 5001`）
- **TC-2.5 SLA 验证**（含 `sla/check-compliance/:ticketId`）
- **TC-2.6 事件 escalate/acknowledge/resolve/close/convert-to-problem**（回归 escalate Bug）
- **TC-2.7 变更 submit/assign/approve/start/complete**（回归审批断裂）
- **TC-2.9 CMDB 关系 + database CI**（回归 `relationships 1001` + database ciTypeId=2）
- **TC-2.10 服务目录与服务请求**（含 `provision/execute` 链路）
- **TC-2.11 知识库 CRUD + RAG 检索**

输出：`stage2_core_data.json`，跨阶段共享 ID 写入 `ids_index.json`

### 阶段 3 — BPMN 工作流深度（M5，45 分钟，本次核心）

完整链路：定义 → 版本 → 部署 → 实例启动 → 任务执行 → 完成/挂起/终止。

- **TC-3.1-3.3 流程定义 CRUD + 版本管理**（定义/导出/克隆/激活/回滚/对比）
- **TC-3.4 实例生命周期**（启动/挂起/恢复/终止/变量设置）
- **TC-3.5 用户任务**（assign/claim/complete/cancel/variables）
- **TC-3.6 `/workflow/*` vs `/bpmn/*` 路由对比**（HTTP 方法不一致问题）
- **TC-3.7 SLA 集成**（`dashboard/sla/violations`、`bottlenecks`、`stats/*`）
- **TC-3.8 监控仪表盘**（metrics/health/audit-logs）
- **TC-3.9 process-trigger 业务绑定**
- **TC-3.10 会签与投票**

输出：`stage3_bpmn.json`

### 阶段 4 — Marketplace + AI（M6+M7，25 分钟）

- **TC-4.1 Marketplace 全流程**（**P0：先确认表缺失，再写迁移并验证 items/install/uninstall/config 链路**）
- **TC-4.2-4.3 AI 分诊/摘要/RAG 检索**（依赖 LLM/Ollama 凭证，可能降级）

### 阶段 5 — 辅助模块（M8，20 分钟）

仪表盘（5+ 端点）、用户/角色/权限/菜单、报表、审计日志、系统配置。

输出：`stage5_auxiliary.json`

### 阶段 6 — 前端白屏诊断 + 回归（15 分钟）

- **TC-6.1 白屏诊断**：agent-browser 截图 `_next/static/chunks/*.js` 404 状态、Console 错误。尝试 `rm -rf .next && npm run dev` 验证 hot reload 修复。
- **TC-6.2 上次 `deep-business-test-report-2026-06-18.md` 8 个回归点** 逐条结论（已修复/部分修复/未修复/引入回归）
- **TC-6.3 上次 `frontend-ux-review-2026-06-19.md` P0-1~P0-4 修复状态**
- **TC-6.4 上次 `browser-e2e-test-report-2026-06-18.md` CSRF 修复回归**

输出：`stage6_regression.md`、`frontend_white_screen_diagnosis.md`

## 报告结构

`docs/review/browser-functional-test-report-2026-06-20.md`：

1. **测试总览**（端点覆盖、通过率、P0/P1/P2 分布、时长）
2. **各模块测试结果**（M0-M8 表格 + 关键 Bug 摘要 + 与上次回归对比）
3. **BPMN 工作流深度专项**（9 个子节）
4. **完整问题清单**（BUG-001 ~ BUG-NNN，含严重度/模块/复现步骤/期望/实际/影响文件/修复建议）
5. **改进建议（按 ROI 优先级）**
   - P0（阻塞开源）：前端白屏 + Marketplace 表 + next.config 严格化 + 根营销页替换
   - P1（1-2 周）：cookie 命名统一 + HTTP 方法统一 + 工单工作流字段回归 + 386 处 console 清理
   - P2（2-4 周）：i18n / Onboarding / 自动化 E2E / Marketplace seed UI
6. **回归分析**：上次三份报告共 16+ 个回归点逐条结论
7. **综合评分**（按 API 完整性、BPMN、Marketplace、前端、回归修复率维度）
8. **附录**（脚本路径、原始数据、关键 ID）

## 风险与限制

| 限制 | 替代方案 |
|------|---------|
| 前端 dev server 白屏（JS 404） | 浏览器仅用于诊断，**不测 UI 交互** |
| BPMN 可视化设计器（拖拽） | 仅测 BPMN XML 导入导出接口 |
| 看板/日历/图表渲染 | 仅测返回数据 API |
| AI 流式响应 | 仅测非流式 POST |
| Marketplace 表缺失 | 阶段 4 中编写并执行迁移脚本 |
| pgvector 扩展可能未装 | 验证 RAG 降级到关键字搜索 |

**明确不重复**：14 模块 UI 视觉评估（上次 e2e）、营销页 UX 分析（frontend-ux-review）、ServiceNow 对标（servicenow-benchmark）、商业就绪度（commercial-readiness）。

## 验证方式

测试结束的标准：

1. 6 个阶段原始数据全部产出（`output/functional-test/stage{0..6}_*`）
2. 主报告 `docs/review/browser-functional-test-report-2026-06-20.md` 写入完整 8 节
3. 所有 P0 问题都有"复现步骤 + 影响文件 + 行号 + 修复建议"
4. 上次 3 份报告的回归点都有明确结论
5. Marketplace 迁移脚本就绪（如缺失）
6. Token 缓存和 ID 索引文件保留，便于下一轮回归测试复用
