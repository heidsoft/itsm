## 二、本轮发现并修复的真实 Bug

### 1. 后端 Bug (5 项)

#### 1.1 [DB-Missing] `tickets` 表缺 `resolution_category` 列
- **现象**: `POST /tickets/workflow/resolve` 报 `pq: column "resolution_category" does not exist`
- **影响**: 工单无法解决，整个解决流程卡死
- **修复**: 迁移 `20260609_add_resolution_category.sql` 新增列
- **文件**: `itsm-backend/migrations/20260609_add_resolution_category.sql`

#### 1.2 [DB-Missing] `tickets` 表缺 `closed_at` 列
- **现象**: `POST /tickets/workflow/close` 报 `pq: column "closed_at" does not exist`
- **影响**: 工单无法关闭
- **修复**: 同一迁移文件追加 `closed_at TIMESTAMP WITH TIME ZONE`
- **文件**: `itsm-backend/migrations/20260609_add_resolution_category.sql`

#### 1.3 [DB-Missing] 缺 `change_approvals` 表
- **现象**: `POST /changes/:id/approvals` 报 `pq: relation "change_approvals" does not exist`
- **影响**: 变更审批完全不可用
- **修复**: 迁移 `20260609_create_change_approvals.sql` 创建表 + 4 个索引
- **文件**: `itsm-backend/migrations/20260609_create_change_approvals.sql`

#### 1.4 [CRITICAL] `/api/v1/analytics/tickets` 返回 500
- **现象**: `failed to group by status: sql/scan: columns do not match (2 > 1)`
- **根因**: `GroupBy(...).Aggregate(Count()).Strings(ctx)` 错用扫描方法
  - `Strings()` 期望单列，但 GROUP BY+COUNT 返回两列（status + count）
- **影响**: 前端工单分析页面无数据
- **修复**: 改用 `Scan(ctx, &[]StatusGroup{})` 自定义结构体扫描
- **文件**: `itsm-backend/service/analytics_service.go`

#### 1.5 [HIGH] `/api/v1/menus` 返回 400
- **现象**: `无效的租户ID` (Invalid tenant ID)
- **根因**: 路由 `/api/v1/menus` 没有 `tenant_id` 路径参数，但 6 个 controller 方法都从 `ctx.Param("tenant_id")` 读取
- **影响**: 菜单管理 API 全部 400
- **修复**: 改用 `ctx.Get("tenant_id")`（auth middleware 已注入）
- **文件**: `itsm-backend/controller/menu_controller.go`

### 2. 前端 Bug (3 项)

#### 2.1 [HIGH] `/approvals` 顶层页面 404
- **现象**: 访问 `/approvals` 返回 404
- **影响**: 用户无法进入审批中心首页
- **修复**: 创建 `app/(main)/approvals/page.tsx`（318 行），含 3 个 Tab + 4 个统计卡片

#### 2.2 [CRITICAL] 前端 API 代理拦截登录请求
- **现象**: `POST /api/v1/auth/login` 返回 401 `Unauthorized: authentication required`
- **根因**: SEC-1 修复后代理强制要求 JWT，但登录接口本身需要在无 JWT 时调用
- **影响**: 前端完全无法登录（直接 API 调用可以）
- **修复**: 新增 `PUBLIC_PATHS` 白名单（login/register/refresh/health/connectors 等）

#### 2.3 [HIGH] 容器构建产物陈旧
- **现象**: Docker 容器内 `.next` 是 6/7 构建的，新页面（如 `/approvals`）未生效
- **影响**: 任何新增前端页面在重启前不可用
- **修复**: 本地 `next build` 后用 `docker cp` 同步到容器并重启
