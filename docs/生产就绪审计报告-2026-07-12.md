> 审计时间：2026-07-12
> 审计范围：按模块检查系统功能，确认已有功能能否生产部署、是否存在"部署麻烦 / 功能不稳定"风险
> 审计方式：源码静态核查 + 编译/类型检查实测 + 部署脚本与 Compose 实测分析 + 历史缺陷文档（部署问题总结 / Bug 清单 v1.1 / 系统测试报告 v1.0）对照

## 一、核心结论（TL;DR）

**已有功能"能"生产部署，但部署链路上存在真实可靠性缺陷，已在本轮修复 4 处。** 用户反馈的"部署麻烦、功能不稳定"主要根因有两类：

1. **部署可靠性缺陷（已修复）**：生产构建会因 GOPROXY 在国内超时；生产 Dockerfile 内存限制过紧易 OOM；部署脚本的 `dc()` 把 docker 真实退出码吞掉，导致构建/校验失败被静默放过、可能把坏镜像部署上去。
2. **反馈来自"未重新部署的旧版本"的概率很高**：历史 Bug 清单（2026-06-29）记录的 6 个 P0 阻塞 Bug，经当前源码核查**绝大部分已修复**（知识库创建、用户角色、审批/SLA 路由 404、工单空描述、服务请求 catalogId 命名等均已在代码中修正）。即"功能不稳定"很可能是旧包在线上，而非当前 main 分支代码本身。

**建议动作**：① 用当前 main 重新走 `deploy-prod.sh` 部署（本轮已加固脚本）；② 补齐 44 个无 controller 测试的模块测试；③ 治理 goroutine 取消传播（288 处 `context.Background()`）。

## 四、"部署麻烦"根因与已修复项

### 4.1 生产构建在国内会超时（已修复）— 对照部署文档"问题二"
- **根因**：`itsm-backend/Dockerfile.prod` 第 7 行 `ARG GOPROXY=https://proxy.golang.org,direct`。在国内部署机拉取 Go 模块会 `dial tcp proxy.golang.org:443: i/o timeout`。dev 版 Dockerfile 已用 `goproxy.cn`，但 prod 版仍是官方源，不一致。
- **已修复**：改为 `ARG GOPROXY=https://goproxy.cn,direct`（海外可 `--build-arg` 覆盖）。

### 4.2 生产构建内存限制过紧、易 OOM（已修复）
- **根因**：`Dockerfile.prod` 设 `GOFLAGS=-p=1 GOMAXPROCS=1 GOGC=20 GOMEMLIMIT=1GiB`。在 2–4GB 规格的 VPS 上构建 2241 文件 + ent 代码生成，极易 OOM 或极慢。dev 版用 `-p=2` 无内存上限。
- **已修复**：放宽为 `-p=2 GOMAXPROCS=2 GOGC=50 GOMEMLIMIT=2GiB`，显著降低小规格主机构建失败率。

### 4.3 部署脚本吞掉 docker 真实退出码（已修复）— 这是"功能不稳定"的隐藏元凶
- **根因**：`scripts/lib/common.sh` 的 `dc()` 在普通模式下把 docker 输出管道给过滤 `while` 循环，**函数返回的是循环退出码（恒为 0）**，把 `docker` 的真实退出码吞掉。叠加 `set -o pipefail`，`build ... 2>&1 | tail -5` 也以 `tail` 的 0 收尾。结果：`if ! dc ...; then 报错中止` 这类判断全部静默通过，**构建失败、Compose 校验失败都拦不住**，可能把编译坏的镜像/配置部署上去——用户体感就是"部署完功能不对/不稳定"。
- **已修复**：重写 `dc()`，先 `out=$(cmd 2>&1); rc=$?` 捕获真实退出码，过滤输出仅用于显示，最后 `return $rc`。实测 `dc` 包裹失败命令现在正确返回非零。

### 4.4 docker compose v1/v2 命令矛盾（已修复）
- **根因**：`dc()` 已做自动探测（有 `docker-compose` v1 则用 v1，否则 v2），但 `deploy-prod.sh` 的 preflight 与构建命令**硬编码 `docker compose`（仅 v2）**。在只装 v1 的机器上，preflight 先报"Docker Compose v2 not available" 而 abort，可它自带的 `dc()` 本能用 v1——自相矛盾，正是"部署麻烦"的典型卡点。
- **已修复**：`deploy-prod.sh` 全部 `docker compose ` → `dc `；`deploy-dev.sh` preflight 同步改为 `dc`，日志提示改为"v1/v2 auto-detected"。

### 4.5 三套 Compose 服务集合不一致（建议，未改）
- `docker-compose.yml`（默认）：含 `itsm-ai-service`，**缺 minio、nginx**，且 `ITSM_ALLOW_ALL_ORIGINS=true`（不安全 dev CORS）。
- `docker-compose.dev.yml`：用 `profiles: [dev]`，含 minio/ollama/prometheus/grafana。
- `docker-compose.prod.yml`：含 minio/nginx，**缺 `itsm-ai-service`**。
- **风险**：用户直接 `docker compose up`（最直觉的命令）会拿到不安全的 dev 配置且缺组件；生产 compose 缺 AI 服务（见 4.6）。建议统一以 `deploy-prod.sh` / `deploy-dev.sh` 为唯一入口，并在各 compose 头部加注释说明"禁止裸跑"。

### 4.6 生产 Compose 缺失 Python `itsm-ai-service`（已确认非阻断，建议统一）
- 经源码核查：Go 后端 **完全不 HTTP 调用** Python `itsm-ai-service`（grep `ai-service|8000|AIService` 零命中）。后端 AI 能力是**进程内** `ai.NewService`（在 `internal/bootstrap/app.go` 构造，含 ragService/triageService 等）。
- 因此生产 compose 缺失该微服务**不构成核心 AI 功能阻断**。但为消除"服务集合不一致"困惑，建议在文档中明确：核心 AI 为进程内实现；若需独立的 Python `/triage`、`/rca`、`/risk` 端点，再显式加入生产 compose。

### 4.7 直接 `docker compose up` 因必填变量而失败（提示，未改）
- 生产 compose 对 `DB_PASSWORD`、`REDIS_PASSWORD`、`JWT_SECRET`、`MINIO_*` 使用 `${VAR:?...}` 必填占位符，必须由 `deploy-prod.sh`（自动 `--env-file .env.prod`）注入。若用户绕过脚本直接跑，会因变量缺失报错。
- **建议**：保持脚本为唯一入口（已加固）；如需裸跑，文档注明先 `cp .env.prod .env` 或 `docker compose --env-file .env.prod -f docker-compose.prod.yml up -d`。

| **残留容器干扰** | 部署文档"问题四"：`KeyError: 'ContainerConfig'` / 容器名冲突 | 已在 `deploy-dev.sh reset` 清理；生产建议同样加强制清理 |
