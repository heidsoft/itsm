# 国内企业ITSM工作流设计方案

## 设计理念

参考黑湖小工单等国内SaaS产品，结合国内企业ITSM最佳实践，强调:

- **快速响应**: P0故障5分钟内启动
- **多级升级**: L1→L2→L3→事件指挥室
- **审批分级**: 低风险快速审批，高风险多级会签
- **即时通知**: 微信/短信/邮件多渠道

## 一、紧急故障响应流程 (P0/P1)

### 流程特点

- **SLA**: P0故障30分钟响应，4小时解决
- **升级路径**: L1→L2→L3→事件指挥室
- **通知机制**: 微信/短信/邮件即时通知

### 流程节点

```
开始(P0上报)
    ↓
[阶段1] 即时通知 (5分钟内)
    - 通知: 服务台主管、一线、运维经理
    - 渠道: 短信+微信+邮件
    ↓
[阶段2] 一线响应 (30分钟内)
    - 人员: l1_support
    - 判断: 可解决/需升级
    ↓
    ├─ 可解决 → 一线修复 → 通知相关方 → 关闭
    │
    └─ 需升级 → [阶段3] L2诊断 (60分钟内)
                    ↓
            ├─ L2解决 → 通知相关方 → 关闭
            │
            └─ L2无法解决 → [阶段4] L3专家处理 (240分钟内)
                                ↓
                        ├─ 解决 → 通知相关方 → 关闭
                        │
                        └─ 重大事件 → 事件指挥室 (480分钟)
                                          ↓
                                     关闭
```

### 关键配置

| 参数 | 值 | 说明 |
|------|-----|------|
| P0故障响应时间 | 5分钟 | 即时通知触发 |
| 一线响应SLA | 30分钟 | L1处理时限 |
| L2诊断SLA | 60分钟 | L2处理时限 |
| L3专家SLA | 240分钟 | 专家处理时限 |
| 重大事件阈值 | 480分钟未解决 | 自动升级至指挥室 |

## 二、标准变更流程 (RFC)

### 流程特点

- **风险分级**: 低/中/高 三级
- **审批层级**: 根据风险等级动态调整
- **回滚计划**: 高风险变更必须准备

### 流程节点

```
开始(变更申请)
    ↓
[阶段1] 变更经理初审 (4小时内)
    ↓
[风险判定网关]
    │
    ├─ 低风险 → 运维经理审批(8小时) → 实施(4小时) → 验证(2小时) → 关闭
    │
    ├─ 中风险 → 技术评审(24小时) → 运维经理审批(12小时) → 实施(8小时) → 验证(4小时) → 关闭
    │
    └─ 高风险 → 专家评审(48小时) → 安全检查(24小时) → 运维总监审批(24小时) → IT总监审批(24小时) → 实施(24小时) → 验证(8小时) → 关闭
```

### 风险等级定义

| 风险等级 | 影响范围 | 审批层级 | 预计周期 |
|---------|---------|---------|---------|
| 低风险 | 单个系统/服务 | 运维经理 | 1-2天 |
| 中风险 | 多个系统/核心服务 | 技术评审+运维经理 | 3-5天 |
| 高风险 | 全局/核心业务 | 专家+安全+总监多层 | 7-14天 |

### 关键配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 低风险阈值 | 业务影响<4小时 | 自动审批 |
| 中风险阈值 | 业务影响4-24小时 | 需技术评审 |
| 高风险阈值 | 业务影响>24小时 | 多级会签 |
| 回滚计划必填 | 高风险变更 | 必须准备 |

## 三、服务请求流程

### 流程特点

- **服务目录化**: IT服务/权限申请/软件/硬件
- **自动化派单**: 根据类型自动分配
- **满意度评价**: 完成后自动发送评价

### 流程节点

```
开始(服务请求提交)
    ↓
[智能分类] (AI自动识别请求类型)
    ↓
[类型判定网关]
    │
    ├─ IT服务 → 一线处理(4小时) → L2处理(8小时) → 通知 → 评价 → 关闭
    │
    ├─ 权限申请 → 主管审批(4小时) → 自动授权 → 通知 → 评价 → 关闭
    │
    ├─ 软件申请 → License检查(4小时)
    │                ↓
    │         ├─ 有License → 自动部署(8小时) → 通知 → 评价 → 关闭
    │         │
    │         └─ 无License → 采购申请 → 等待(7天) → 部署 → 关闭
    │
    └─ 硬件申请 → 审批(8小时) → 备货(5天) → 交付 → 通知 → 评价 → 关闭
```

### 关键配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 标准服务SLA | 4小时 | IT服务响应 |
| 权限申请SLA | 4小时 | 自动审批通过 |
| 软件部署SLA | 8小时 | 有License时 |
| 硬件备货SLA | 5天 | 包含物流时间 |
| 满意度调查 | 72小时 | 自动发送 |

## 五、通知机制

### 通知渠道优先级

| 优先级 | 渠道 | 适用场景 |
|-------|------|---------|
| P0 | 短信>微信>电话 | 紧急故障 |
| P1 | 微信>短信>邮件 | 重要事件 |
| P2 | 邮件>站内消息 | 一般通知 |
| P3 | 站内消息 | 低优先级 |

### 通知触发条件

| 触发事件 | 通知对象 | 渠道 |
|---------|---------|------|
| 新P0故障 | 一线/L2/运维经理 | 短信+微信 |
| 故障升级L2 | L2/运维经理 | 微信 |
| 故障升级L3 | L3/运维总监/IT总监 | 短信+微信+电话 |
| 重大事件 | IT总监/运维总监 | 短信+电话 |
| 变更审批 | 审批人 | 邮件+微信 |
| 变更完成 | 申请人 | 邮件 |
| 服务请求关闭 | 申请人 | 邮件+微信 |
