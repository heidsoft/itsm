# Workflow 控制台诊断与功能设计

## 目标

`/workflow` 不是单纯的列表页，而应该成为 ITSM 的工作流控制台入口，承担：

- 流程定义管理
- 流程设计与版本治理
- 实例运行监控
- 任务与会签处理
- 模板复用与 AI 创建入口

这条主线要围绕国内企业常见的 ITILv3 场景展开，例如事件、变更、服务请求、问题管理、发布审批。

## 当前诊断结论

### 1. 页面语义偏散

当前 `/workflow` 已经具备列表、创建、导入、部署、版本和设计入口，但这些能力分散在多个页面和弹窗中，缺少一个明确的“控制台”心智模型。

### 2. 详情入口语义不准确

列表中的“查看”按钮过去直接跳到设计器，用户会误以为这是只读详情，但实际进入的是编辑语境。

### 3. 版本与 BPMN 数据链路不完整

设计器页依赖 `bpmn_xml` 和版本 XML，但 API 映射不完整时，已有流程会打开成空白模板，影响编辑和回滚。

### 4. 分类值存在语义污染风险

创建/编辑流程时如果直接把中文文案写入 category，后续筛选、统计和自动化规则会被本地化文本污染，应该改成稳定编码。

### 5. 本地过滤与分页存在体验割裂

当前列表页的筛选是前端本地筛选，而分页来自后端，容易出现“总数和当前列表不一致”的感知问题。

## 推荐信息架构

### 顶层导航

- 控制台概览
- 流程定义
- 设计器
- 实例监控
- 版本管理
- 模板中心
- 自动化与 AI
- 审计与报表

### 流程定义页应该提供的核心动作

- 新建流程
- 从模板创建
- 导入 BPMN
- 设计
- 发布/激活
- 暂停/停用
- 查看实例
- 查看版本
- 导出定义

### 详情页应该提供的核心信息

- 流程名称、Key、分类、版本、状态
- BPMN 是否存在
- 当前实例数、运行中实例数
- 创建人、更新时间
- 快捷跳转到设计器、实例页、版本页

## 目标设计

### 1. 控制台首页

建议增加一个工作流首页卡片区，展示：

- 流程定义总数
- 运行中实例
- 今日新建实例
- 待处理任务
- SLA 风险流程
- 最近变更流程

同时补一个“快捷入口面板”，直接跳转到：

- 监控仪表盘
- 实例监控
- 版本治理
- 自动化规则
- 审计追踪
- 审批设计器

### 2. 流程定义列表

列表作为主入口，但操作要分层：

- 主操作：新建、设计、部署
- 次操作：查看、复制、导出、版本、实例
- 危险操作：删除、停用

### 3. 设计器页

设计器应该支持：

- 打开现有 BPMN XML
- 版本切换
- 保存草稿
- 保存并发布
- 节点属性编辑
- AI 生成流程骨架

### 4. 版本页

版本页建议强化为“版本治理”：

- 版本对比
- 激活
- 回滚
- 版本备注
- 变更日志

### 5. 实例页

实例页建议强化为“运行态治理”：

- 按流程、状态、时间过滤
- 查看当前节点
- 查看任务流转
- 暂停/恢复/终止
- 查看异常原因

## AI Native 演进方向

建议把 AI 能力放在三个位置：

1. 流程创建时
- 用自然语言生成 BPMN 初稿
- 根据场景推荐审批节点和 SLA

2. 流程设计时
- 根据当前节点提示下一步分支
- 识别缺失的异常路径、回退路径、通知节点

3. 运行治理时
- 自动摘要实例异常
- 自动生成处置建议
- 自动识别瓶颈节点

## 下一阶段建议

### Phase 1

- 修复 BPMN XML 映射
- 统一流程分类编码
- 让“查看”和“设计”语义分离
- 增加详情弹窗或详情页

### Phase 2

- 补控制台首页
- 增加指标卡、趋势图、风险清单
- 统一实例、版本、设计器导航

### Phase 3

- 接入 AI 生成流程
- 接入飞书、企业微信、钉钉通知
- 建立连接器市场、Skill 市场、插件市场

## 结论

当前 `/workflow` 已具备成为企业级工作流中心的基础，但还缺少统一语义、完整数据链路和控制台视图。

这次修正后，页面会更接近“工作流控制台”而不是“工作流列表页”，后续可以继续向流程治理平台演进。
