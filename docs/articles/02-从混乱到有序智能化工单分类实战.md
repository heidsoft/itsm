# 从混乱到有序：AI驱动的智能化工单分类实战

## 一文讲透工单智能分诊技术架构、算法原理与落地经验

---

工单分类是IT运维中最基础却最重要的工作。

一个工单进入系统后，首先需要判断它的类型（事件？请求？投诉？）、优先级（紧急？高？中？低？）、归属团队（一线？二线？开发？运维？），这些判断决定了工单的处理路径和质量标准。

听起来简单，做起来却极其消耗人力。

某互联网公司IT负责人曾告诉我，他们团队10个人，每天处理200多个工单，其中光分类判断就要消耗3个人力。"不是技术活，但特别费神。一天下来脑子都是木的。"

这是行业普遍痛点。而AI正在彻底改变这个局面。

本文将以ITSM开源项目为例，深入剖析智能工单分类的技术架构、算法原理和实战经验。无论你是技术决策者还是一线工程师，都能从中获得有价值的参考。

## 第一部分：为什么工单分类如此重要

### 1.1 分类错误的后果

工单分类不只是一道"选择题"，它直接影响整个处理流程。

**分错了类型，处理路径就错了。** 一个安全事件被误判为普通请求，转到了普通工程师手里而不是安全团队，可能导致安全问题被延迟处理。一个性能问题被当作功能缺陷处理，可能在错误的团队之间转来转去，浪费大量时间。

**分错了优先级，后果更严重。** 紧急故障被判断为低优先级，可能在黄金处理时间内无人响应，导致故障蔓延。系统可用性 SLA被触发，业务损失随之而来。一次P0故障的处理延迟，可能造成百万级别的业务损失。

**分错了归属团队，工单就会在团队之间踢皮球。** 接收团队觉得不是自己负责，婉拒并转派；下一个团队同样处理。一来二去，工单在错误的人手里转了三四圈才找到正确的处理人，时间已经浪费了一大半。

这些场景在每个IT团队都真实发生过，根源往往就是分类这一个环节。

### 1.2 人工分类的困境

既然分类这么重要，为什么人工分类质量难以保证？

**第一，分类规则复杂。** 成熟的IT组织往往有几十甚至上百种工单类型，每种类型有对应的处理流程、责任团队、SLA要求。这些规则靠人脑记忆，本身就是负担。更何况规则本身也在不断变化——新系统上线、旧系统退役、业务调整——分类规则需要持续更新。

**第二，描述千人千面。** 用户提交工单时，他们的描述方式差异巨大。同样是"系统登录不了"，有人写"OA系统登不上去"，有人写"钉钉扫码没反应"，有人写"报错500"。AI需要理解这些五花八门的表达背后的真实意图。

**第三，边界情况难以判断。** 大多数工单分类清晰，但总有20%的边界情况让工程师纠结。"这个算事件还是请求？""这个影响程度算高还是中？"不同人可能有不同答案。

**第四，疲劳导致错误。** 分类是高频重复工作，工程师在处理了大量工单之后，注意力下降，分类错误率上升。特别是在夜班、加班等场景，疲劳影响更明显。

### 1.3 AI分类的优势

AI在分类任务上有独特的优势：

**速度快。** 人工分类一个工单平均需要3-5分钟，AI只需要毫秒级。这不只是效率差异，更关键的是AI可以7x24小时保持稳定性能，不会疲劳、不会情绪波动、不会因为加班而错误率上升。

**一致性好。** 同一个边界case，AI每次都会给出相同判断（只要模型不变）。人工分类则可能因为状态不同、判断标准理解差异而出现摇摆。AI让分类标准真正一致。

**可解释。** 现代AI分类系统不只是给出结果，还能提供判断依据——为什么这么分？参考了哪些特征？这让复核和优化变得简单。

**持续学习。** AI模型可以从历史数据中学习，持续优化分类准确率。随着处理工单数量增加，模型越来越懂你的业务，分类越来越准。

## 第二部分：智能分诊的技术架构

### 2.1 双层架构设计

ITSM的智能分诊系统采用**关键词+语义理解**的双层架构。

为什么需要两层？

**单一关键词匹配太粗糙。** 规则引擎响应快、准确率高，但对于表述多样的用户描述无能为力。用户可能用100种方式描述"密码问题"，你不可能写出100条规则覆盖所有表达。

**纯语义理解太慢太贵。** 大语言模型理解能力强，但每次调用有延迟和成本。如果所有工单都走LLM，成本会非常高。更重要的是，对于明确的常见问题，用规则引擎处理更高效。

双层架构的核心思想是：**简单问题用规则，复杂问题用语义**。

```
工单提交
    │
    ▼
┌─────────────────┐
│  第一层：关键词匹配  │ ← 毫秒级响应，零成本
│  (Keyword Match) │
└────────┬────────┘
         │
    命中规则？
         │
    ┌────┴────┐
    │         │
   Yes        No
    │         │
    ▼         ▼
┌───────┐ ┌─────────────────┐
│ 直接  │ │  第二层：语义理解   │
│ 分类  │ │ (LLM Inference) │
└───┬───┘ └────────┬────────┘
    │               │
    │         ┌────┴────┐
    │        High      Low
    │     Confidence  Confidence
    │         │           │
    │         ▼           ▼
    │    ┌────────┐  ┌─────────┐
    │    │ 自动   │  │ 人工   │
    │    │ 执行   │  │ 复核   │
    │    └────────┘  └─────────┘
    │
    ▼
返回分类结果
```

### 2.2 关键词匹配层

关键词匹配是整个系统的兜底层。它的工作原理是：维护一个"关键词→分类"的映射表，当工单中出现特定词汇时，触发对应的分类动作。

```go
// 关键词匹配规则示例
var keywordRules = []Rule{
    {
        Keywords:    []string{"密码", "登录不上", "账户锁定", "改密"},
        Category:    "账号问题",
        Priority:    "high",
        Team:        "identity-team",
        Confidence:  0.95,
    },
    {
        Keywords:    []string{"网络", "连不上", "ping", "不通"},
        Category:    "网络问题",
        Priority:    "medium",
        Team:        "network-team",
        Confidence:  0.90,
    },
    {
        Keywords:    []string{"慢", "卡顿", "超时", "响应慢"},
        Category:    "性能问题",
        Priority:    "medium",
        Team:        "performance-team",
        Confidence:  0.85,
    },
}
```

关键词匹配的优势是**零幻觉、零成本**。只要命中规则，结果就是确定的，不会出现"AI胡编"的担忧。对于明确的常见问题，关键词匹配是第一选择。

实现细节上，需要考虑：

**同义词扩展**。用户写"上不去"和"登录不了"表达的是同一个意思，规则需要覆盖这些变体。我们维护了一个中文同义词库，把意思相近的词汇归为一组，只要命中一组中的任一词汇就算匹配。

**词序无关**。匹配时不考虑关键词出现的顺序，只要求工单内容中包含这些词汇即可。

**多关键词组合**。有些分类需要多个关键词同时出现才能确定。比如"数据库连接数爆满"需要"数据库"和"连接"和"满"三个词同时出现，防止"数据库备份导致的连接短暂满载"被误判为异常情况。

**优先级设计**。规则之间可能冲突，比如"网络"和"网卡"都匹配不同的分类。我们用优先级权重来解决，高优先级规则优先匹配。

### 2.3 语义理解层

当关键词匹配无法确定分类时，系统会调用大语言模型进行语义理解。

语义理解层的核心是一个精心设计的提示词（Prompt）：

```
你是一个IT工单分类助手。你的任务是根据工单内容，判断工单的类型、优先级和推荐处理团队。

## 工单内容
{工单标题}
{工单描述}

## 可选类型
- incident: 故障/异常事件
- request: 服务请求/一般性请求
- problem: 问题/需要根因分析的复杂情况
- change: 变更相关
- complaint: 投诉/服务不满意

## 优先级定义
- critical: 核心业务系统故障，影响大量用户，需要立即处理
- high: 非核心但影响明显的问题，需要尽快处理
- medium: 一般性问题，常规流程处理
- low: 次要问题，可安排处理

## 处理团队
- first-line: 一线支持（处理简单问题、答疑）
- second-line: 二线支持（技术问题处理）
- development: 研发团队（代码/功能问题）
- operations: 运维团队（基础设施、发布）
- security: 安全团队（安全相关问题）

请返回JSON格式的分类结果，包含：
1. category: 工单类型
2. priority: 优先级
3. team: 推荐团队
4. confidence: 置信度(0-1)
5. reason: 判断理由（1-2句话）
```

这个Prompt的设计有几个要点：

**Few-shot示例**。在Prompt中包含几个工单分类示例，帮助模型理解任务要求。这比纯文字描述效果更好。

**结构化输出**。要求模型返回JSON格式，便于程序解析。置信度字段让后续处理有明确依据。

**判断理由**。要求模型给出判断理由，这对人工复核非常重要。当分类结果需要人工确认时，理由能帮助工程师快速判断是否正确。

**中文优化**。Prompt针对中文语境优化，使用中文术语和表达习惯。

### 2.4 置信度与人工复核

置信度是连接AI和人工的桥梁。

AI对每次判断给出一个置信度分数（0到1之间）。分数越高，说明AI越有把握。当置信度高于阈值（比如0.85）时，系统自动执行分类结果；当置信度较低时，系统会标记为"待复核"，由人工最终确认。

```go
// 置信度阈值配置
const (
    AutoConfirmThreshold = 0.85  // 高于此阈值自动执行
    ManualReviewThreshold = 0.60  // 低于此阈值标记为困难工单
)

// 分类决策逻辑
func MakeClassificationDecision(result *ClassificationResult) Action {
    if result.Confidence >= AutoConfirmThreshold {
        return AutoExecute
    }
    if result.Confidence >= ManualReviewThreshold {
        return ManualReview
    }
    return FallbackToKeyword  // 置信度过低，回退到规则引擎
}
```

这套机制的效果是：**高频简单问题完全自动化，低频复杂问题保留人工判断**。日常80%的工单由AI自动处理，只有20%的边界case需要人工介入。

人工复核的结果同时会反馈给模型，用于持续优化后续判断。

## 第三部分：算法实现细节

### 3.1 关键词匹配算法

关键词匹配看似简单，实现起来有若干技术细节值得关注。

**高效匹配**。我们使用Aho-Corasick自动机算法实现多模式字符串匹配。这个算法可以在O(n)时间复杂度内完成所有关键词的匹配，其中n是文本长度。与简单的逐个关键词正则匹配相比，性能提升显著。

**中文分词**。中文文本需要先分词再匹配关键词。我们使用结巴分词库进行中文分词，并配合停用词过滤去除"的、了、是"等无意义词汇。

```go
// 关键词匹配实现
func MatchKeywords(text string, rules []Rule) *RuleMatch {
    // 1. 中文分词
    words := jieba.Cut(text, false)

    // 2. 构建词集合（去重）
    wordSet := make(map[string]bool)
    for _, word := range words {
        if !isStopWord(word) {
            wordSet[word] = true
        }
    }

    // 3. 规则匹配
    for _, rule := range rules {
        matched := countMatchedKeywords(wordSet, rule.Keywords)
        if matched >= rule.MinMatchCount {
            return &RuleMatch{
                Rule:      rule,
                Matched:   matched,
                Total:     len(rule.Keywords),
                Precision: float32(matched) / float32(len(rule.Keywords)),
            }
        }
    }

    return nil
}
```

**同义词映射**。分词后，我们会把同义词映射到统一的标准词。比如"密码问题"、"账号问题"、"登录不了"都映射到标准词"account_issue"。这样规则只需要针对标准词编写，匹配效率更高。

### 3.2 语义分类模型

语义理解层我们选择调用大语言模型API实现，而非自己训练模型。原因有几点：

**效果更好**。通用大模型在语义理解能力上远超垂直领域的小模型。即使是简单任务，模型的泛化能力也让规则编写更轻松。

**成本可控**。对于工单分类这个场景，80%的问题被关键词层处理，只有20%需要调用LLM。按量计费的API成本非常低。

**迭代更快**。Prompt改一改就能测试新效果，不像训练模型需要数据准备、训练、调参的漫长周期。

目前系统支持对接多种LLM提供商：

```go
// LLM Provider抽象
type LLMProvider interface {
    Complete(prompt string) (string, error)
}

// 支持的提供商
var providers = map[string]LLMProvider{
    "openai":   &OpenAIProvider{},
    "claude":   &ClaudeProvider{},
    "ollama":   &OllamaProvider{},    // 本地部署
    "dashscope": &DashScopeProvider{}, // 阿里云
}
```

对于数据敏感的企业，可以选择部署Ollama本地模型，数据完全不外传。对于追求效果的场景，OpenAI的GPT-4或Anthropic的Claude是更好的选择。

### 3.3 性能优化

实际生产环境中，工单量可能非常大。性能优化是系统设计的重要考量。

**异步处理**。关键词匹配是同步的，毫秒级完成。语义理解因为涉及API调用，有几百毫秒延迟。我们将语义分类设计为异步任务，不阻塞工单创建流程。工单创建后立即返回，分类结果通过回调或轮询获取。

```go
// 异步分类处理
func (s *TriageService) ClassifyAsync(ticket *Ticket) {
    // 立即返回，工单创建不受影响
    go func() {
        result := s.ClassifySync(ticket)
        s.saveResult(ticket.ID, result)

        // 置信度低于阈值，触发人工复核通知
        if result.NeedManualReview() {
            s.notifyReviewer(ticket, result)
        }
    }()
}
```

**批量处理**。对于需要语义理解的工单，系统会积累一批（默认10个或等待500ms），然后批量发送给LLM API。这减少了API调用次数，降低了成本和延迟。

**缓存优化**。相似的工单内容可能重复出现。我们缓存了最近1000个工单的分类结果（按内容hash索引），命中缓存时直接返回，不调用任何模型。

### 4.2 常见问题与解决

**问题一：分类结果不符合预期**

这种情况通常是因为Prompt不够精确，或者模型对业务场景理解有偏差。

解决方法：增加Few-shot示例，在Prompt中包含更多与你的业务相关的分类样例。观察错误分类的case，针对性地补充示例。

**问题二：置信度普遍偏低**

可能是因为工单描述不规范，或者业务边界比较模糊。

解决方法：优化Prompt中的类型定义，让边界更清晰。同时可以考虑在工单提交时增加必填字段，帮助AI更好地理解上下文。

**问题三：LLM调用成本太高**

说明关键词匹配覆盖率不够，大量工单走了语义理解层。

解决方法：分析哪些类型的工单频繁触发LLM调用，针对这些类型增加关键词规则。目标是让80%以上的工单被关键词层处理。

**问题四：模型幻觉导致错误分类**

大模型偶尔会"编造"理由或给出不存在的分类。

解决方法：设置更低的自动执行阈值，或者对LLM返回结果做合法性校验。检查返回的分类是否在定义的类型列表中，不合法则降级到人工复核。

### 4.3 持续优化机制

AI分类系统不是一次性工程，需要持续优化。

**定期Review**。每周抽检一批AI分类结果，分析错误case，更新规则或Prompt。

**用户反馈**。当处理工程师修改了AI分类结果时，这个反馈会被记录，用于后续优化。

**数据积累**。积累足够多的标注数据后，可以考虑微调专属模型，进一步提升准确率。

**规则迭代**。随着业务发展，旧规则可能不再适用。需要定期审视规则库，删除过时规则，补充新规则。

## 第五部分：代码示例与集成指南

### 5.1 快速集成

ITSM的分类服务已经封装为独立模块，可以方便地集成到现有系统。

**引入依赖**：

```go
import "github.com/heidsoft/itsm/service/triage"
```

**创建服务实例**：

```go
config := &triage.Config{
    LLMProvider:  "openai",
    APIKey:       os.Getenv("OPENAI_API_KEY"),
    AutoThreshold: 0.85,
}

svc, err := triage.NewService(config)
if err != nil {
    log.Fatal(err)
}
```

**调用分类**：

```go
ticket := &triage.Ticket{
    Title:       "OA系统登录失败",
    Description: "用户反映早上开始无法登录OA系统，点击登录按钮后报错'会话超时'",
}

result, err := svc.Classify(ticket)
if err != nil {
    log.Error(err)
    return
}

fmt.Printf("分类结果: %s\n", result.Category)
fmt.Printf("优先级: %s\n", result.Priority)
fmt.Printf("推荐团队: %s\n", result.Team)
fmt.Printf("置信度: %.2f\n", result.Confidence)
```

### 5.2 自定义规则配置

规则配置通过YAML文件管理，方便调整：

```yaml
# config/triage-rules.yaml

      - 网络

      - 丢包

```

### 5.3 监控与告警

生产环境建议配置监控告警，及时发现分类异常：

```go
// 分类质量监控
metrics := &triage.Metrics{
    TotalClassified:    counter("triage.total"),
    AutoExecuted:      counter("triage.auto"),
    ManualReview:       counter("triage.manual_review"),
    FallbackToKeyword:  counter("triage.fallback"),
    AvgConfidence:      histogram("triage.confidence"),
    LLM调用延迟:         histogram("triage.llm_latency"),
}

// 配置告警规则
alerts := []triage.Alert{
    {
        Condition: "manual_review_rate > 0.30",  // 复核率超过30%告警
        Message:   "AI分类复核率异常，请检查分类规则或模型配置",
    },
    {
        Condition: "avg_confidence < 0.70",  // 平均置信度低于0.7告警
        Message:   "AI分类置信度下降，建议review最近的分类错误case",
    },
}
```
