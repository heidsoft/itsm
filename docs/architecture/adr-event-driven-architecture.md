# ADR-002: 引入事件驱动架构

## Status
**Proposed** (提议中)

## Context

当前系统存在以下问题：

1. **同步调用耦合**：工单创建时，SLA 计算、审批触发、通知发送都在同一请求中同步执行，导致：
   - 主流程响应时间长
   - 任意环节失败都会导致整个请求失败
   - 难以独立扩展各处理环节

2. **服务边界模糊**：100+ service 文件混在一起，新增功能难以定位

3. **扩展性受限**：未来难以支持：
   - 多端通知（邮件/企微/短信）
   - 异步 AI 分诊
   - 审计日志
   - 跨系统集成

## Decision

引入**事件驱动架构**，采用**模块化单体 + 事件总线**模式：

1. **新增 `service/common/event/` 基础设施**：
   - 事件定义 (`event.go`): 定义领域事件接口和具体事件
   - 事件总线 (`event_bus.go`): 内存实现（开发/小规模）
   - Redis 实现 (`redis_event_bus.go`): 生产环境高可用

2. **改造现有服务**：
   - 核心操作（Create/Assign/Resolve）发布领域事件
   - 耗时操作（AI分诊、SLA、通知）改为异步订阅处理

3. **渐进式改造策略**：
   - 新功能直接使用事件驱动
   - 旧服务逐步迁移
   - 保持接口兼容

## Consequences

### 正面
- **解耦**：核心流程与扩展功能独立
- **性能**：主流程响应时间显著缩短
- **可扩展**：新增通知渠道/处理逻辑无需修改核心代码
- **可观测**：事件日志便于追踪和审计
- **容错**：各处理环节独立失败不影响主流程

### 负面
- **复杂度**：需要引入消息队列基础设施
- **最终一致性**：事件处理是异步的，部分场景需要考虑状态同步
- **调试难度**：分布式追踪要求更高
- **开发门槛**：团队需要理解事件驱动模式

### 需要考虑
- 事件幂等处理（防止重复消费）
- 事件丢失处理（消息队列持久化）
- 事件顺序性（同一工单的事件需有序处理）

## 实施计划

| 阶段 | 时间 | 任务 |
|:---|:---:|:---|
| Phase 1 | 1周 | 创建事件基础设施（接口 + 内存实现） |
| Phase 2 | 1周 | 改造 TicketService.CreateTicket 为事件驱动 |
| Phase 3 | 1周 | 实现 SLA/审批/通知处理器 |
| Phase 4 | 2周 | 引入 Redis Stream 实现（生产环境） |
| Phase 5 | 持续 | 迁移其他服务（Incident/Change/Problem） |
